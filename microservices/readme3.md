Задание 3.
Задача 1.
В качестве решения для развёртывания, запуска и управления приложениями предлагается использовать Kubernetes как основную платформу оркестрации контейнеров.

Обоснование выбора:
Kubernetes является стандартом де-факто для эксплуатации микросервисных систем. Он изначально разработан для работы с контейнеризованными приложениями и предоставляет встроенные механизмы управления их жизненным циклом, масштабирования, отказоустойчивости и сетевого взаимодействия. Решение хорошо масштабируется, не привязано к конкретному облачному провайдеру и интегрируется с современными инструментами CI/CD, мониторинга и логирования.

Архитектура решения:
Приложения упаковываются в Docker-образы и хранятся в контейнерном реестре. Kubernetes разворачивает их в виде Pod и управляет ими с помощью следующих объектов:
- Deployment — для stateless-сервисов
- StatefulSet — для stateful-сервисов (например, базы данных, очереди, кеши)
- Service — для внутреннего взаимодействия сервисов
- Ingress Controller — для маршрутизации внешнего трафика

Конфигурация и секреты выносятся из контейнеров и управляются отдельно.

Соответствие требованиям:
1. Поддержка контейнеров:
- Kubernetes работает с контейнерами (например, Docker) как с базовой единицей запуска
- Каждый микросервис разворачивается как отдельный контейнер, что обеспечивает изоляцию и независимость сервисов
2. Обнаружение сервисов и маршрутизация запросов:
- Внутреннее обнаружение сервисов реализуется через объект Service
- Kubernetes автоматически создаёт DNS-имена для сервисов внутри кластера
Внешняя маршрутизация осуществляется через Ingress Controller (например, NGINX или Traefik):
- принимает входящие HTTP/HTTPS-запросы
- направляет их в соответствующие сервисы по правилам (домен, путь и т.д.)
3. Горизонтальное масштабирование:
- Каждый сервис может быть запущен в нескольких экземплярах (репликах)
- Kubernetes автоматически распределяет нагрузку между ними
4. Автоматическое масштабирование:
- Используется Horizontal Pod Autoscaler (HPA)
- Масштабирование происходит по метрикам (CPU, память, пользовательские метрики)
Дополнительно в облаке используется Cluster Autoscaler:
- автоматически увеличивает или уменьшает количество узлов
5. Разделение внутренних и внешних ресурсов:
- Внутренние сервисы публикуются с типом ClusterIP
- Внешний доступ осуществляется через Ingress или LoadBalancer
Дополнительно применяется NetworkPolicy:
- ограничивает сетевое взаимодействие между сервисами
Таким образом достигается разделение:
- внешний слой — Ingress / API Gateway
- внутренний слой — сервисы, базы данных, кеши и очереди
6. Конфигурация и безопасное хранение секретов:
- Конфигурация приложений передаётся через ConfigMap
- Данные доступны как переменные среды или файлы
Чувствительные данные:
- хранятся в Kubernetes Secret
- для production рекомендуется использовать HashiCorp Vault или облачные Secret Manager
Преимущества такого подхода:
- отсутствие хранения секретов в коде
- централизованное управление
- поддержка ротации и аудита
Дополнительные возможности:
- Namespace — для разделения окружений (dev, stage, prod)
- ResourceQuota и LimitRange — для ограничения ресурсов
- Rolling Update — для обновлений без простоя
- Liveness и Readiness пробы — для контроля состояния приложений
Выводы:
- Kubernetes полностью удовлетворяет требованиям задачи
- обеспечивает контейнеризацию, сервис-дискавери, маршрутизацию
- поддерживает масштабирование и изоляцию ресурсов
- позволяет безопасно управлять конфигурацией и секретами

Задача 2.
Требуется развернуть Redis Cluster, состоящий из трёх шард (master-узлов) и трёх реплик.
Архитектура кластера:
Кластер состоит из:
- 3 master (шарды)
- 3 replica (реплики)
Распределение по виртуальным машинам:
VM1:
- master shard 1
- replica shard 2
VM2:
- master shard 2
- replica shard 3
VM3:
- master shard 3
- replica shard 1

Обоснование схемы:
- Реплика каждого master-узла размещается на другой виртуальной машине
- Это обеспечивает отказоустойчивость
При выходе из строя одной VM:
- реплика автоматически повышается до master
- кластер продолжает работу
Принцип работы Redis Cluster:
- Данные распределяются по 16384 hash slots
- Слоты делятся между master-узлами
- Каждый master отвечает за свою часть данных
Репликация:
- реплики синхронизируются со своими master
- используются для отказоустойчивости
Работа клиента:
- клиент может подключиться к любому узлу
- при необходимости получает перенаправление на нужный shard
Конфигурация узлов:
- cluster-enabled yes
- cluster-config-file nodes.conf
- cluster-node-timeout 5000
- appendonly yes
Необходимые порты:
- 6379 — клиентские подключения
- 16379 — cluster bus
Создание кластера:
redis-cli --cluster create [ip1:6379](ip1:6379) [ip2:6379](ip2:6379) [ip3:6379](ip3:6379) [ip4:6379](ip4:6379) [ip5:6379](ip5:6379) [ip6:6379](ip6:6379) --cluster-replicas 1
Параметр:
--cluster-replicas 1 — одна реплика на каждый master
Преимущества решения:
- горизонтальное масштабирование
- отказоустойчивость
- распределение нагрузки
- подходит для хранения сессий и временных данных

Ограничения:
- multi-key операции ограничены одним shard
- требуется поддержка Redis Cluster со стороны приложения

Выводы:
- Redis Cluster из 3 master и 3 replica обеспечивает
- отказоустойчивость
* масштабируемость
* эффективное хранение сессионных данных
