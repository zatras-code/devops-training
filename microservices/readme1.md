Задача 1. API Gateway

В микросервисной архитектуре нужен API Gateway — это точка входа, через которую идут все запросы к сервисам. Он должен уметь направлять запросы в нужный сервис, проверять авторизацию и работать с HTTPS.

Я рассмотрел несколько популярных решений: Kong, Apache APISIX, Traefik и NGINX.

Если кратко:

Kong — полноценный API Gateway с готовыми плагинами для аутентификации, логирования и других задач.  
APISIX — похож на Kong, но более гибкий и быстрый, при этом чуть сложнее в настройке.  
Traefik — простой вариант, хорошо подходит для Docker и Kubernetes, но с ограниченными возможностями по авторизации.  
NGINX — надёжный, но для полноценной работы как API Gateway часто требует дополнительной настройки или платной версии.

В итоге я бы выбрал Kong.

Почему:

- он полностью закрывает все требования задачи;
- поддерживает маршрутизацию "из коробки";
- есть готовые решения для аутентификации;
- умеет работать с HTTPS;
- при этом достаточно простой в эксплуатации.

Вывод: для роли API Gateway лучше всего подходит Kong Gateway.

Задача 2. Брокер сообщений

Для связи между микросервисами нужен брокер сообщений. Он позволяет сервисам обмениваться данными асинхронно, не завися друг от друга.

Требования: надёжность, хранение сообщений, высокая скорость, разграничение доступа и простота в использовании.

Я рассмотрел несколько вариантов: RabbitMQ, Kafka, NATS и Pulsar.

Коротко по ним:

RabbitMQ — классический брокер, простой в использовании, поддерживает много протоколов.  
Kafka — очень быстрый и мощный, хорошо подходит для обработки больших потоков данных, но сложнее в эксплуатации.  
NATS — лёгкий и быстрый, но не такой универсальный.  
Pulsar — мощный, но довольно сложный в настройке и поддержке.

Я бы выбрал RabbitMQ.

Почему:

- есть кластеризация и отказоустойчивость;
- сообщения можно хранить на диске;
- хорошая производительность;
- поддерживает разные протоколы (это удобно для интеграций);
- можно гибко настраивать доступ;
- проще в эксплуатации, чем Kafka или Pulsar.

Вывод: оптимальный вариант — RabbitMQ.

------------------------------------------------------------

Общий итог

Для системы на микросервисах я бы предложил:

API Gateway — Kong  
Брокер сообщений — RabbitMQ  

Такое решение даёт нормальный баланс между возможностями, надёжностью и простотой поддержки.
