Задание 2.
Задача 1. Обеспечение процесса разработки

Для организации процесса разработки предлагается использовать следующий стек: GitLab SaaS, GitLab CI/CD, GitLab Runner (self-hosted), HashiCorp Vault и GitLab Container Registry.

Каждый микросервис размещается в отдельном Git-репозитории в GitLab. Разработчики работают через механизм merge request. При событиях в системе контроля версий, таких как push, создание merge request или тегов, автоматически запускаются пайплайны GitLab CI/CD. Также предусмотрена возможность ручного запуска сборки через интерфейс с передачей параметров.

Сборки выполняются с помощью GitLab Runner. Часть runner'ов может быть размещена в облаке, а часть — на собственных серверах компании. Это позволяет гибко управлять ресурсами и выполнять сборки внутри защищённой инфраструктуры.

Секретные данные не хранятся в коде. Они передаются в пайплайн либо через защищённые переменные GitLab, либо через HashiCorp Vault, который выдаёт временные токены. После завершения сборки артефакты и Docker-образы сохраняются в GitLab Container Registry.

Решение соответствует требованиям следующим образом. Используется облачная система — GitLab SaaS. Система контроля версий основана на Git. Для каждого сервиса создаётся отдельный репозиторий. Сборки запускаются по событиям из системы контроля версий, а также могут запускаться вручную с указанием параметров. Настройки могут быть привязаны к каждой сборке через переменные, правила и конфигурацию пайплайна. Поддерживаются шаблоны сборок с помощью include и extends. Секретные данные хранятся безопасно через Vault или защищённые переменные. В одном репозитории можно реализовать несколько конфигураций сборки. Поддерживаются кастомные шаги. Можно использовать собственные Docker-образы. Агенты сборки можно развернуть на собственных серверах. Поддерживается параллельный запуск сборок и тестов.

Выбор данного решения обусловлен тем, что GitLab объединяет систему контроля версий и CI/CD в одной платформе, что упрощает внедрение и сопровождение. Он хорошо масштабируется и широко применяется в микросервисных архитектурах.

Задача 2. Организация сбора и анализа логов

Для логирования предлагается использовать Fluent Bit, Kafka, OpenSearch и OpenSearch Dashboards.

Все сервисы пишут логи в stdout и stderr. На каждом хосте работает агент Fluent Bit, который собирает эти логи, добавляет метаданные и отправляет их в Kafka. Kafka выступает промежуточным слоем, который обеспечивает буферизацию и повышает надёжность доставки. Это позволяет избежать потери логов при временных сбоях хранилища.

Из Kafka логи считываются и индексируются в OpenSearch. OpenSearch хранит данные и обеспечивает быстрый поиск и фильтрацию. Для работы пользователей используется OpenSearch Dashboards, где можно выполнять поиск, строить запросы и сохранять их.

Решение соответствует требованиям следующим образом. Логи собираются централизованно со всех хостов с помощью Fluent Bit. Приложения не требуют изменений и пишут в stdout. Надёжность доставки обеспечивается за счёт Kafka и буферизации. OpenSearch предоставляет мощные возможности поиска и фильтрации. OpenSearch Dashboards даёт пользовательский интерфейс, где разработчики могут самостоятельно работать с логами. Также есть возможность сохранять поисковые запросы и делиться ссылками на них.

Данное решение выбрано, так как оно хорошо масштабируется, устойчиво к сбоям и удобно для микросервисной архитектуры. Fluent Bit минимально нагружает систему, Kafka повышает надёжность, а OpenSearch предоставляет удобный инструмент анализа.

Задача 3. Организация мониторинга

Для мониторинга предлагается использовать Prometheus, Node Exporter, cAdvisor или kubelet metrics, а также Grafana.

На каждом хосте устанавливается Node Exporter, который собирает метрики состояния системы: загрузку CPU, использование памяти, дисков и сети. Для контейнеров и сервисов используются метрики cAdvisor или kubelet, что позволяет отслеживать потребление ресурсов на уровне сервисов.

Каждый сервис при необходимости публикует собственные метрики в формате Prometheus, например количество запросов, время ответа и число ошибок. Prometheus регулярно опрашивает все источники метрик и сохраняет данные.

Grafana используется как интерфейс для работы с метриками. В ней можно строить графики, выполнять запросы и создавать дашборды для отслеживания состояния системы.

Решение соответствует требованиям следующим образом. Метрики собираются со всех хостов с помощью Node Exporter. Метрики CPU, RAM, дисков и сети доступны из коробки. Метрики по сервисам собираются через cAdvisor или kubelet. Специфичные метрики публикуются самими сервисами. Prometheus и Grafana предоставляют интерфейсы для запросов, анализа и построения панелей мониторинга.

Выбор данного решения обусловлен тем, что Prometheus и Grafana являются стандартом де-факто в микросервисной архитектуре. Они хорошо масштабируются, имеют развитую экосистему и позволяют эффективно отслеживать как инфраструктуру, так и бизнес-метрики.

Итог
Предложенный стек включает GitLab для разработки и CI/CD, Fluent Bit, Kafka и OpenSearch для логирования, а также Prometheus и Grafana для мониторинга. Это решение закрывает все требования задачи, масштабируется и подходит для эксплуатации в облачной среде с возможностью использования собственных серверов.
