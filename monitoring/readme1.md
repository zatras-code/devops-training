1. Минимальный набор метрик для мониторинга.
Метрики HTTP:
- количество запросов (RPS)
- время ответа (latency)
- коды ответов (2xx, 4xx, 5xx)
Почему:
позволяет понять нагрузку на сервис, скорость работы и наличие ошибок.

Метрики CPU:
- загрузка CPU
- Load Average
Почему:
вычисления нагружают процессор, поэтому важно контролировать его загрузку.

Метрики памяти:
- использование RAM
- доступная память
Почему:
при нехватке памяти приложение может замедляться или падать.

Метрики диска:
- свободное место на диске
- disk I/O
- количество inode
Почему:
система сохраняет отчёты на диск, поэтому диск может заполниться или возникнут проблемы с записью файлов.
Disk usage и disk I/O

2. Что предложить менеджеру продукта.
Менеджеру обычно важны метрики качества сервиса, а не инфраструктурные показатели.
SLI/SLO метрики:
- Доступность сервиса (Availability)
- Процент успешно обработанных запросов - Показывает насколько сервис доступен для клиентов
- Время ответа сервиса (например: p50, p95, p99) - показывает насколько быстро сервис отвечает пользователям
- Количество обработанных запросов - показывает нагрузку и использование сервиса
- Процент ошибок - процент запросов, завершившихся ошибкой

3. Нет системы сбора логов, но разработчикам нужны ошибки.
Возможные варианты:
- Использовать Sentry (приложение отправляет ошибки напрямую в систему отслеживания ошибок) - разработчики сразу видят stacktrace, не требует полноценного log storage
- Отправлять ошибки как метрики
Приложение может экспортировать метрику: app_errors_total
Тогда система мониторинга сможет:
строить графики ошибок
отправлять алерты разработчикам.
- Сохранять ошибки в файл и дать разработчикам доступ

4. Почему SLA не поднимается выше 70%
Формула:
summ_2xx_requests / summ_all_requests
При этом в системе нет кодов 4xx и 5xx.
Ошибка в том, что не учитываются ответы 3xx (redirect).
HTTP коды делятся на:
1xx — информационные
2xx — успешные
3xx — перенаправления
4xx — ошибка клиента
5xx — ошибка сервера
Коды 3xx попадают в summ_all_requests, но не попадают в 2xx.
Поэтому процент успешных запросов сильно занижен.

5. Плюсы и минусы push и pull мониторинга
Pull модель
Система мониторинга сама опрашивает сервисы и забирает метрики.
Плюсы:
- простой контроль частоты опроса
- легче масштабировать
- система мониторинга знает, какие таргеты существуют
- проще обнаруживать недоступные сервисы
Минусы
- сложно мониторить краткоживущие процессы
- могут быть проблемы с NAT или firewall

Push модель
Сервисы сами отправляют метрики в систему мониторинга.
Плюсы
- подходит для краткоживущих задач
- работает в закрытых сетях
- проще отправлять метрики из разных сред
Минусы
- сложнее контролировать поток метрик
- система мониторинга не знает, какие источники должны существовать
- возможна перегрузка сервера мониторинга

6. Какие системы push/pull
- Prometheus
Основная модель — pull (scrape метрик).
Но есть Pushgateway, поэтому можно считать гибридом.

- TICK (Telegraf + InfluxDB)
Модель push — агент Telegraf отправляет метрики.

- Zabbix
Гибридная система:
agent (push)
agentless checks (pull)

- VictoriaMetrics
Чаще используется с Prometheus (pull),
но может принимать push-метрики.
Поэтому гибридная модель.

- Nagios
Классическая модель pull (active checks).
Но есть passive checks - гибрид.
